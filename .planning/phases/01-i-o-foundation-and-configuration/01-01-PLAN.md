<feature>
  <name>Hook I/O Utilities (hook-utils.js)</name>
  <files>hooks/hook-utils.js, test/hook-utils.test.js</files>
  <behavior>
    readStdinJSON:
    - Given valid JSON piped to stdin -> returns parsed object
    - Given empty stdin (no pipe, EOF) -> returns {}
    - Given malformed JSON on stdin -> returns {}
    - Given JSON with hook_event_name field -> parsed object includes that field

    buildOutput:
    - Given input with hook_event_name and no fields -> returns { hookSpecificOutput: { hookEventName: <from input> } }
    - Given input with hook_event_name and additionalContext in fields -> returns envelope with additionalContext inside hookSpecificOutput
    - Given fields with hookEventName override -> uses override instead of input.hook_event_name
    - Given no hookEventName in input or fields -> defaults to "PostToolUse"
    - Given extra fields (decision, reason) -> copies them into hookSpecificOutput

    resolveProjectRoot:
    - Given CLAUDE_PROJECT_DIR env var set -> returns path.resolve(CLAUDE_PROJECT_DIR)
    - Given no CLAUDE_PROJECT_DIR -> returns path.resolve(cwd)
  </behavior>
  <implementation>
    After tests are written and confirmed RED:

    hooks/hook-utils.js implementation per 01-RESEARCH.md code examples:
    - readStdinJSON: try { raw = fs.readFileSync(0, 'utf8').trim(); if (!raw) return {}; return JSON.parse(raw); } catch { return {}; }
    - buildOutput: construct hookSpecificOutput envelope, auto-detect hookEventName from input.hook_event_name, copy all fields except hookEventName into hookSpecificOutput
    - resolveProjectRoot: check process.env.CLAUDE_PROJECT_DIR first, fall back to process.cwd(), always path.resolve()
    - module.exports = { readStdinJSON, buildOutput, resolveProjectRoot }
    - CommonJS, "use strict", no external dependencies

    Testing readStdinJSON requires subprocess approach since fd 0 cannot be mocked in-process.
    Use execFileSync to run a small test script that requires hook-utils and calls readStdinJSON,
    piping JSON via input option. For empty stdin test, pipe empty string.
  </implementation>
</feature>
