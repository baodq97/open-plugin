# How Might We (HMW) Patterns

## Format

```
HMW-{SEQ}: How might we [verb] [subject] so that [outcome]?
```

## Derivation Rules

1. **From POV**: Every POV statement generates 2-5 HMW questions
2. **From Pain Points**: High-severity PPs (>= 7) should have dedicated HMWs
3. **Scope check**: If a HMW can be answered with a single feature, it's too narrow. If it would take years to solve, it's too broad.

## Pattern Library

### Amplify the Good
- "How might we **amplify** {existing positive behavior} to {benefit}?"
- Example: "How might we amplify the workaround Sarah uses to share files so it becomes the default workflow?"

### Remove the Bad
- "How might we **eliminate** {pain point} for {persona}?"
- Example: "How might we eliminate the 3-step manual copy process that wastes 2 hours weekly?"

### Explore the Opposite
- "How might we make {frustrating task} **enjoyable** for {persona}?"
- Example: "How might we make error handling enjoyable for developers?"

### Question Assumptions
- "How might we {achieve goal} **without** {assumed constraint}?"
- Example: "How might we onboard new users without requiring a tutorial?"

### Change the Status Quo
- "How might we **rethink** {current process} to {new outcome}?"
- Example: "How might we rethink the approval workflow to reduce turnaround from 3 days to 3 hours?"

### Reframe the Constraint
- "How might we **turn** {constraint} into an **advantage**?"
- Example: "How might we turn the limited budget into a feature that enforces simplicity?"

## Quality Checks

- Each HMW must be answerable with multiple solutions (not one obvious answer)
- Each HMW must be specific enough to inspire concrete ideas
- Each HMW must trace back to a POV statement or high-severity pain point
- Avoid HMW questions that are really requirements in disguise ("How might we add a search bar?")
